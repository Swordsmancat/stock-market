# Performance Optimization and Data Display Fix

## 目标

解决首页加载缓慢、市场筛选切换慢、前端数据不显示的问题，并实现多数据源自适应支持。

## 背景

### 当前问题
1. **页面加载慢** - 首页打开需要 3.34 秒
2. **切换选项慢** - 市场筛选下拉框切换响应慢
3. **前端数据不显示** - 后端有数据（7/10 个指数有数据），但前端可能渲染有问题
4. **数据源单一** - 当前仅使用 yfinance，未充分利用多数据源能力

### 技术现状
- 后端 API 响应时间: 3.34秒 (过慢)
- 数据库有数据: AAPL 635条，标普500 627条
- 已配置的数据源:
  - ✅ yfinance (当前使用)
  - ✅ tushare (已配置 token)
  - ✅ akshare (已启用)
  - ✅ mock (测试用)

## 需求

### 高优先级

#### 1. 优化 API 响应速度

**目标**: 将首页 API 响应时间从 3.34秒降低到 <500ms

**技术方案**:
- 添加 Redis 缓存层
- 缓存 `/dashboard/market-overview` 响应
- TTL: 1-5分钟 (可配置)
- 缓存键: `dashboard:market-overview:{provider}:{date}`

**实现细节**:
```python
# 伪代码
@cache(ttl=300, key_pattern="dashboard:market-overview:{provider}:{date}")
async def get_market_overview_payload(...):
    ...
```

#### 2. 修复前端数据显示

**问题诊断**:
- 后端有数据但前端不显示
- 可能是数据格式不匹配或渲染错误

**检查项**:
- Console 错误日志
- 网络请求响应
- 数据结构映射
- React 组件渲染

#### 3. 优化前端性能

**React 性能优化**:
- 使用 `React.memo` 包装 `MarketTicker` 组件
- 优化状态更新，避免不必要的重渲染
- 使用 `useMemo` 缓存过滤后的数据

**代码分割**:
- 按需加载图表组件
- 懒加载非关键 UI 组件

#### 4. 实现多数据源自适应

**自动降级策略**:
1. **主数据源优先** - 默认使用配置的数据源
2. **失败自动切换** - 主数据源失败时切换到备用源
3. **混合数据源** - 不同市场使用最佳数据源
   - CN 市场: tushare / akshare
   - HK 市场: yfinance
   - US 市场: yfinance / alpha vantage

**配置示例**:
```json
{
  "data_sources": {
    "CN": ["tushare", "akshare", "yfinance"],
    "HK": ["yfinance", "akshare"],
    "US": ["yfinance", "alpha_vantage"]
  },
  "fallback_enabled": true,
  "cache_ttl": 300
}
```

### 中优先级

#### 5. 添加数据刷新机制

- 手动刷新按钮
- 自动刷新 (可配置间隔: 1-10分钟)
- 显示数据更新时间

#### 6. 性能监控

- 添加 API 响应时间监控
- 前端性能指标 (FCP, LCP, TTI)
- 数据源健康状态检查

### 低优先级

#### 7. WebSocket 实时推送

- 对于需要实时数据的场景
- 替代轮询机制
- 减少服务器负载

## 验收标准

### 性能指标
- [ ] 首页 API 响应时间 < 500ms (缓存命中)
- [ ] 首次加载 < 2秒 (无缓存)
- [ ] 市场筛选切换响应 < 100ms
- [ ] 首屏渲染时间 (LCP) < 2.5秒

### 功能验证
- [ ] 前端正确显示所有有数据的指数
- [ ] 市场筛选器正常工作 (全部/CN/HK/US)
- [ ] 数据刷新机制工作正常
- [ ] 主数据源失败时自动降级到备用源
- [ ] 缓存正确更新和过期

### 兼容性
- [ ] 现有功能不受影响
- [ ] 多数据源切换透明 (用户无感知)
- [ ] 降级时记录警告日志

## 技术约束

### 缓存策略
- 使用现有的 Redis 实例
- 缓存键命名规范: `{namespace}:{resource}:{params}`
- 支持手动清除缓存

### 数据源配置
- 数据源优先级可配置
- 支持动态启用/禁用数据源
- 保留现有的 provider 参数兼容性

### 性能基准
- 数据库查询 < 100ms
- 缓存读取 < 10ms
- API 响应 (缓存命中) < 500ms

## 不在范围内

- 完整的实时 WebSocket 实现 (留待后续)
- 前端性能持续监控平台 (可后续集成)
- 复杂的数据源路由逻辑 (保持简单)

## 开放问题

1. **缓存失效策略** - 主动失效 or TTL 被动过期？
   - 推荐: TTL + 手动失效 API
   
2. **数据源切换粒度** - 按市场 or 按标的？
   - 推荐: 按市场配置，按标的实际执行
   
3. **前端缓存** - 是否需要浏览器端缓存？
   - 推荐: 先实现服务端缓存，前端暂用 SWR

4. **监控告警** - 性能下降时如何通知？
   - 推荐: 先记录日志，后续集成告警

## 成功指标

- 首页加载时间减少 80%+
- 用户体验评分提升
- 数据源故障时服务仍可用
- 99% 请求在 1 秒内完成
