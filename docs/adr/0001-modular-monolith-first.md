# ADR 0001: 第一阶段采用模块化单体架构

## 状态

Accepted

## 背景

股票分析平台需要同时覆盖数据采集、技术指标、基本面分析、新闻舆情、AI 报告、模拟组合和 Web Dashboard。系统天然具备未来拆分服务的边界，但第一阶段的主要目标是验证完整业务闭环，而不是追求分布式架构复杂度。

平台第一阶段还存在多个需要快速迭代的边界：

- 多市场证券主数据模型。
- 行情、财报、新闻数据源适配。
- 技术指标和分析特征口径。
- AI 报告输入、引用和风险提示格式。
- 模拟组合顾问的责任边界。

如果一开始采用微服务，会在业务模型尚未稳定时引入服务发现、跨服务事务、接口版本治理、部署编排、分布式追踪和消息一致性问题。

## 决策

第一阶段采用模块化单体架构：

- 后端 API、异步任务、领域模型、数据源适配、分析模块和 AI 模块在一个后端代码库内开发。
- 通过清晰目录边界隔离模块职责。
- 后台耗时任务通过 Celery/Redis 执行。
- 数据库采用 PostgreSQL + TimescaleDB。
- 前端 Web Dashboard 独立为 Next.js 应用。

推荐代码边界：

| 路径 | 职责 |
|---|---|
| `apps/api` | HTTP API、认证、权限、Dashboard 查询接口 |
| `apps/worker` | 异步任务入口，执行采集、指标计算、新闻抓取和报告生成 |
| `apps/web` | Next.js Web Dashboard |
| `packages/domain` | 市场、标的、行情、新闻、报告、组合等领域模型 |
| `packages/providers` | 行情、财报、新闻、宏观数据源适配器 |
| `packages/analytics` | 技术指标、基本面指标、新闻舆情、组合风险计算 |
| `packages/ai` | LLM Provider、提示词模板、报告生成、引用组装 |
| `packages/shared` | 配置、日志、错误类型、通用工具 |

## 备选方案

### 方案 A：一开始采用微服务

优点：

- 每个服务可以独立部署和扩缩容。
- 服务边界更明确。
- 适合团队规模较大、模块负载差异明显的场景。

缺点：

- 初期开发和运维复杂度高。
- 业务模型未稳定时，服务边界容易频繁变动。
- 需要更早处理分布式事务、消息一致性、服务观测和接口治理。

### 方案 B：纯数据平台/湖仓架构优先

优点：

- 适合大规模离线研究、回测和特征工程。
- 数据保留和批处理能力强。

缺点：

- 对 Web Dashboard、报告订阅和交互式分析的产品闭环支持较慢。
- 对第一阶段验证用户价值来说工程投入偏重。

### 方案 C：模块化单体优先

优点：

- 开发、部署和调试成本低。
- 能快速验证数据采集到 AI 报告的主链路。
- 仍可通过目录、接口和数据库边界保留后续拆分能力。

缺点：

- 当采集、新闻 NLP 或 AI 报告负载快速增长时，需要拆分服务。
- 需要团队自律维护模块边界，避免形成大泥球。

## 结果

采用方案 C：模块化单体优先。

当出现以下信号时，再将对应模块拆成独立服务：

- 行情采集吞吐或限流逻辑明显影响 API 响应。
- 新闻 NLP 或情绪分析需要独立扩缩容。
- AI 报告生成耗时长，挤占其他后台任务资源。
- Web API 与后台任务在部署、扩缩容或故障隔离上产生明显冲突。

## 影响

- 第一阶段实施计划以单仓库、模块化目录、异步任务和共享数据库为基础。
- 数据库表必须从第一天保留多市场、多数据源、数据截止时间和引用字段。
- 后续服务拆分时，应优先复用现有模块接口和任务边界，而不是重写业务逻辑。
