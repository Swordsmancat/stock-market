# Internationalization (i18n) Design

## Overview
This document outlines the design for implementing Chinese/English (zh/en) language toggling in the Stock Analysis Platform's Next.js frontend using `next-intl`.

## Architecture: `next-intl` with App Router

We will use `next-intl` to handle both the routing and the translation dictionaries. This is the recommended approach for Next.js 13+ App Router.

### 1. Routing & Middleware
- **Dynamic Segment:** All page routes will be moved under a `[locale]` dynamic segment. For example, `apps/web/app/page.tsx` becomes `apps/web/app/[locale]/page.tsx`.
- **Middleware:** A `middleware.ts` file at the root of `apps/web/src` (or `apps/web`) will intercept incoming requests.
  - If a user visits `/`, they will be redirected to the default locale: `/en`.
  - It will handle locale detection based on browser headers (`Accept-Language`), but default to `en`.

### 2. Translation Dictionaries
Translations will be stored as flat or nested JSON files.
- `apps/web/messages/en.json`
- `apps/web/messages/zh.json`

Example structure:
```json
{
  "Navigation": {
    "dashboard": "Dashboard",
    "watchlist": "Watchlist"
  },
  "Dashboard": {
    "title": "Dashboard",
    "latestPrice": "Latest Price"
  }
}
```

### 3. UI Integration
- **Language Switcher:** A new `LanguageSwitcher` component will be added to the `TopNavBar`, likely as a dropdown menu alongside the Theme toggle. It will allow users to switch between "English" and "中文".
- **Provider:** A `NextIntlClientProvider` will wrap the layout to pass translation messages down to Client Components.
- **Hooks:** Components will use the `useTranslations` hook to fetch localized strings.

## Technology Stack
- **Library:** `next-intl`
- **Framework:** Next.js (App Router)
- **UI:** shadcn/ui `DropdownMenu` for the language switcher.

## Implementation Details

### Directory Structure Changes
```text
apps/web/
├── messages/
│   ├── en.json
│   └── zh.json
├── i18n.ts             # next-intl configuration
├── middleware.ts       # Locale routing middleware
└── app/
    └── [locale]/       # Moved from app/
        ├── layout.tsx
        ├── page.tsx
        └── ...
```

### Data Flow
1. User visits `/portfolios`.
2. `middleware.ts` intercepts, detects missing locale, redirects to `/en/portfolios`.
3. Next.js routes to `app/[locale]/portfolios/page.tsx`.
4. `i18n.ts` loads `messages/en.json`.
5. `NextIntlClientProvider` provides messages to the React tree.
6. Components call `useTranslations('Namespace')` to render text.

## Next Steps for Implementation
1. Install `next-intl`.
2. Create `messages/en.json` and `messages/zh.json` with initial keys for the layout and dashboard.
3. Setup `i18n.ts` and `middleware.ts`.
4. Move existing `app/*` files into `app/[locale]/`.
5. Update `next.config.mjs` to use the `next-intl` plugin.
6. Create the `LanguageSwitcher` component and add it to `TopNavBar`.
7. Refactor `SidebarNavigation`, `TopNavBar`, and `Dashboard` to use `useTranslations`.
